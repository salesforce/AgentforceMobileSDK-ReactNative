# Implementation Plan: Separate Employee Agent and Service Agent Apps

## Context

This plan implements a lightweight approach to package two distinct apps from the same repository:
- **Service Agent App**: Customer-facing, anonymous authentication, no Mobile SDK dependency (~smaller binary)
- **Employee Agent App**: Internal workforce, OAuth authentication, requires Salesforce Mobile SDK

**Why**: The bridge has flexible runtime detection supporting both modes, but different branches have different configurations:
1. **main branch**: No Mobile SDK → Service Agent only
2. **dev/employeeAgent branches**: Mobile SDK added → Both modes available, but always bundled
3. **Problem**: Can't have both app types from the same codebase
   - To ship Employee Agent, must add Mobile SDK (bloats Service Agent builds)
   - To ship lightweight Service Agent, must remove Mobile SDK (loses Employee Agent capability)
   - Can't package them as separate apps with different bundle IDs
   - Can't maintain independent release cycles

**Approach**: Use industry-standard multi-target patterns (xcodegen for iOS, product flavors for Android) to generate two apps from a single shared codebase with maximum code reuse (>98% shared code).

---

## Bridge Architecture: Optional Mobile SDK Support

The bridge is **already designed** to work with or without Mobile SDK at runtime:

### Android Bridge (compileOnly + Runtime Detection)
- **Build**: `compileOnly "com.salesforce.mobilesdk:SalesforceReact:13.1.1"` (line 54 of bridge/build.gradle)
  - Bridge compiles against Mobile SDK classes
  - Doesn't bundle them in the bridge AAR
  - Host app decides whether to provide at runtime

- **Runtime**: Reflection-based detection in `AgentforcePackage.kt`:
  ```kotlin
  private fun isMobileSdkAvailable(): Boolean {
      return try {
          Class.forName("com.salesforce.androidsdk.app.SalesforceSDKManager")
          true
      } catch (e: ClassNotFoundException) {
          false
      }
  }
  ```
  - If Mobile SDK present: registers `EmployeeAgentAuthBridge` (Employee Agent auth support)
  - If Mobile SDK absent: skips registration (Service Agent only)

### iOS Bridge (Subspecs)
- **Core** subspec (default): Service Agent only, no Mobile SDK dependency
- **WithMobileSDK** subspec: Adds `SalesforceSDKCore` dependency for Employee Agent

This design makes the multi-app approach straightforward - the bridge adapts automatically.

### Critical File

**android/app/build.gradle** (UPDATE)

Add after `defaultConfig` block:
```gradle
android {
    namespace = "com.salesforce.android.reactagentforce.app"
    compileSdk rootProject.ext.compileSdk

    defaultConfig {
        applicationId "com.salesforce.android.reactagentforce"
        minSdkVersion rootProject.ext.minSdkVersion
        targetSdkVersion rootProject.ext.targetSdkVersion
        versionCode 1
        versionName "1.0"
    }

    // NEW: Product flavors
    flavorDimensions "agentType"

    productFlavors {
        serviceAgent {
            dimension "agentType"
            applicationIdSuffix ".service"
            versionNameSuffix "-service"
            buildConfigField "String", "AGENT_TYPE", '"SERVICE"'
            resValue "string", "app_name", "Service Agent"
        }

        employeeAgent {
            dimension "agentType"
            applicationIdSuffix ".employee"
            versionNameSuffix "-employee"
            buildConfigField "String", "AGENT_TYPE", '"EMPLOYEE"'
            resValue "string", "app_name", "Employee Agent"
        }
    }

    // ... rest of config
}

dependencies {
    // ... existing dependencies

    // CHANGE: Make Mobile SDK conditional using product flavors
    // CURRENT STATE:
    //   - main branch: No Mobile SDK dependency → Service Agent only
    //   - dev/employeeAgent branches (line 124-125): implementation("com.salesforce.mobilesdk:SalesforceReact:13.1.1")
    //     → Always includes ~5-10MB Mobile SDK
    //     → Bridge detects it's present and enables Employee Agent auth
    //     → Service Agent mode doesn't use it (dead code if only using Service Agent)

    // REMOVE: implementation("com.salesforce.mobilesdk:SalesforceReact:13.1.1")

    // ADD: Flavor-specific dependency
    employeeAgentImplementation("com.salesforce.mobilesdk:SalesforceReact:13.1.1")

    // Result:
    // - Service Agent flavor: Omits Mobile SDK → smaller APK, same as main branch
    // - Employee Agent flavor: Includes Mobile SDK → full functionality
    // - Bridge's runtime detection (AgentforcePackage.kt) adapts automatically
}
```

This creates 4 build variants:
- `serviceAgentDebug` → `com.salesforce.android.reactagentforce.service`
- `serviceAgentRelease` → `com.salesforce.android.reactagentforce.service`
- `employeeAgentDebug` → `com.salesforce.android.reactagentforce.employee`
- `employeeAgentRelease` → `com.salesforce.android.reactagentforce.employee`

## Shared JavaScript Layer

### Approach: Configuration at Build Time

The current React Native code is already well-structured and can be shared 100% between apps. No changes required to core logic.

**Optional Enhancement** (not blocking): Create `src/config/AppConfig.ts` to expose build-time configuration:
```typescript
export const APP_CONFIG = {
  AGENT_MODE: (process.env.AGENT_MODE || 'hybrid') as 'service' | 'employee' | 'hybrid',
  APP_NAME: process.env.APP_NAME || 'Agentforce',
};

export const FEATURES = {
  SHOW_SERVICE_AGENT: APP_CONFIG.AGENT_MODE !== 'employee',
  SHOW_EMPLOYEE_AGENT: APP_CONFIG.AGENT_MODE !== 'service',
};
```

This allows removing UI tabs conditionally if desired, but the current runtime configuration approach also works fine.
